## 工具使用规则

1. 你可以使用一切命令行工具来完成任务，并且我还很鼓励这样做。但是你需要记住一点，对于不幂等的操作或者无法撤销的命令，你需要提前备份，或者是经过我的允许才可以执行。
2. 你被鼓励使用各种工具来从网络获取更多信息
   - 优先考虑官方网页，其次是知名博主写的经验文章，不要从 CSDN 或者各种内容农场获取信息
   - 禁止使用内置的 fetch 和 search 工具，应该使用我们准备的 crawl 工具，使用时可以参考 fetch-and-search skill
   - jina（备用）：仅用于学术搜索（arxiv、ssrn、bibtex）和高级功能（classify、deduplicate、expand_query 等），如果额度受限可以考虑 fetch MCP
3. 未经我的允许，**禁止**使用 git 进行提交或者 push 代码的操作（使用 superpowers 时除外）

## 文风规范

1. 你的文风必须准确精炼，避免使用无关字符，如 emoji。
2. 尽量避免编写额外的文档，如果你认为有必要编写，你**必须**先征得我的同意。
3. 你不可以使用加粗、斜体 markdown 语法规则。
4. 无论用户使用什么语言输入，你都要用中文回复用户

## 注释规范

核心原则：注释解释 WHY，不解释 WHAT。代码本身应该能说明它在做什么，注释的价值在于提供代码无法表达的信息。

该写注释的场景：

1. 设计意图和业务规则 — 字段/函数的行为不是从名字就能看出来的。例如 `allowedUsers` 空数组表示全部禁止，这个语义不显然
2. 默认值来源 — 当默认值在另一个地方定义或不是显而易见时，使用 `@defaultValue` 标签
3. 非显而易见的约束和语义 — 比如 `deleteAfterRun` 不是删除而是自动禁用；`consecutiveErrors` 在 3 次时触发自动禁用；`"missed"` 表示服务器宕机期间错过执行
4. 复杂算法或策略 — 当函数采用多步骤、多策略的复合逻辑时，概述整体流程
5. 跨系统协作关系 — 函数之间的配合关系，如 `snapshotWorkspace` 和 `detectNewFiles` 的前后依赖
6. monkey patch / 测试覆盖点 — 参数存在的目的是方便测试注入
7. 领域概念和专有名词 — 如 Vikunja 是"任务管理器"，不是通用词汇

不该写注释的场景：

1. 重复函数/字段名 — `path: string` 加 `/** Absolute path on disk */` 没有增加任何信息
2. 重复类型签名 — `data: string` 加 `/** The data */` 纯粹是噪音
3. 显而易见的操作 — `getFileList` 返回文件列表，名字已经说清楚了
4. 仪式性注释 — 给每个接口、每个字段都加注释，形成"注释疲劳"，反而让有用的注释被淹没
5. 重复函数签名的文档 — `/** Classify file paths as photo or document based on their extension */` 对比 `function categorizeFiles(filePaths: string[]): DetectedArtifact[]`，前者没有增加任何信息

注释风格：

- `/** */` (TSDoc/JSDoc) 用于导出的公共 API、接口、类型，会被 IDE 和文档工具消费
- `//` 用于实现细节、行内说明
- 不使用 `/* */` 块注释（Google TypeScript Style Guide）
- TODO 格式：`// TODO(owner): description`

## 编码规范

原则：你的代码可能运行在下面这些复杂的平台和环境：

- Node.js 16-22
- Windows 7-11、Mac、Linux(CentOS、Ubuntu)
- x86、ARM 平台
- 同时打开实例
- 工作区内可能有无限多文件，文件大小也可能非常大
- 用户连续操作一周

因此你可能要注意下面的问题：

1. 不同操作系统的文件路径以及不同的规则（比如 macOS 大小写不敏感，Linux 大小写敏感），路径长度可能有限制
2. IO 环境非常复杂，你必须注意下面的情况：
   1. 假设要读写的文件可能不存在、可能无比巨大、可能是个软链接、可能编码不是 UTF-8
   2. 假设要监听、读取的目录下文件无限多、别的软件可能也会修改。尤其是 `.git`、`node_modules` 等
   3. 合并写操作，合并多个小文件操作，减少磁盘 I/O。
   4. 用 stream 记得关闭流
   5. 写文件一定要考虑写竞争问题
   6. 下载的文件默认没有执行权限
   7. 磁盘大小有限，不要无限写
   8. 用户当前的版本不一定是上一个版本，可能跨版本升级。持久存储要做好数据兼容
